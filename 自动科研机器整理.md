# **自动科研机器整理**

## 一、核心要求

### 1. 一个核心目标

- **寻找“性价比最高”的实现：** 寻找实现简单但效果公认较好、能够真正落地的方案，而不是空有噱头。

### 2. 两类关键功能

- **环境与资源管理：** 能够自动化管理沙箱环境（Sandbox）和计算资源。
- **全流程科研覆盖：** 包括文献搜索、代码编写、实验执行、数据分析、论文撰写与编译。

### 3. 三项“好”的评价标准

- **Token 使用效率：** 在保证效果的前提下，消耗的 Token 越少越好（成本意识）。
- **运行稳定性：** 系统能够长时间稳定运行，不轻易崩盘。
- **工程健壮性：** 代码中必须有足够的 **Error Handling（错误处理）**，能够处理实验或调用中的异常。

### 4. 一个架构约束

- **拒绝“个人版”黑盒：** 明确表示不适合 Claude Code 或 OpenCoder 这种面向个人用户的闭源/高度集成形态。
- **拥抱“平台化”架构：** 必须是**后端 API 化**的，支持多用户通过 Web、邮件或通讯软件发送 Request，并能异步查询任务状态。
- **逻辑解耦：** 倾向于借鉴成熟工具的 Skills（技能库）和 Prompt（提示词），但要把 Agent 逻辑替换为自研的 Python Workflow。
