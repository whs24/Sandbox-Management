# 沙箱管理项目最终对比

## 总体排序

如果目标是建设“自动工业工程问题求解科研机器”的沙箱管理层，推荐优先级如下：

1. AutoResearchClaw：最接近完整科研沙箱，适合作为主蓝本。
2. RD-Agent：资源受控 Docker 实验执行器成熟，适合作为执行层参考。
3. deer-flow：SandboxProvider、middleware、AioSandbox 生命周期管理优秀，适合作为 Agent 控制面参考。
4. SWE-agent：持久 shell session、工具协议、repo reset 和 timeout 机制优秀，适合作为代码编辑型 Agent 参考。
5. DeepResearch/Tongyi：外部 Python interpreter 服务适合作为轻量计算工具，不适合作为主实验沙箱。
6. AI-Scientist v2：进程级解释器和 timeout 有参考价值，但需要外层容器。
7. AI-Researcher：持久 Docker + TCP shell 方便，但安全边界弱。
8. Auto-Deep-Research：与 AI-Researcher 类似，且有动态下载命令服务风险。
9. AI-Scientist v1：主要是科研流程，几乎没有内置沙箱。

## 横向对比表

| 仓库 | 沙箱形态 | 主要优点 | 主要风险/缺口 | 对本项目适用度 |
|---|---|---|---|---|
| AutoResearchClaw | 本地/Docker/SSH/Colab/Agentic Docker | 统一协议、Docker 资源限制、网络策略、入口校验、指标解析、partial result | 需加强工业求解器、许可证、集群调度、seccomp/rootless | 高 |
| RD-Agent | DockerEnv/LocalEnv/CondaEnv | Docker SDK、GPU/CPU/内存/shm、日志、缓存、清理 | 缺少 Agent 命令审计、默认网络较宽、不是完整安全框架 | 高 |
| deer-flow | SandboxProvider + Local/Aio/Remote | provider 抽象、middleware、路径映射、warm pool、远端 provisioner | LocalSandbox 非安全边界，Aio 后端资源/网络限制不足 | 中高 |
| SWE-agent | Deployment runtime + bash session | 持久 session、repo reset、工具安装、动作 timeout、连续 timeout 熔断 | Docker 细节在外部依赖，缺少科研实验 metrics/artifacts | 中高 |
| DeepResearch/Tongyi | SandboxFusion 外部 Python interpreter | 窄工具接口、服务化、短 timeout、stdout/stderr 标准化 | 只适合小代码片段，缺少项目/文件/求解器/资源管理 | 中 |
| AI-Scientist v2 | multiprocessing 子进程 interpreter | 事件队列、SIGINT/terminate/kill、ExecutionResult | 无容器边界，进程清理可能误伤，缺少网络/资源/文件隔离 | 中低 |
| AI-Researcher | Docker + TCP command server | 持久容器、流式 JSON 输出、任务级容器命名 | root、host userns、裸端口、无限制 shell、无资源/网络策略 | 中低 |
| Auto-Deep-Research | Docker/LocalEnv + TCP command server | 持久容器、端口锁、浏览器研究环境组合 | 动态下载 tcp_server、root shell、LocalEnv、无资源/网络限制 | 低到中 |
| AI-Scientist v1 | 本地 subprocess + 外部 Docker 建议 | 科研循环清晰，stderr 反馈给 Agent | 无内置沙箱、无隔离、无资源控制 | 低 |

## 按能力维度对比

| 能力 | 表现最好的仓库 | 说明 |
|---|---|---|
| 沙箱抽象 | deer-flow、AutoResearchClaw、SWE-agent | provider/factory/deployment 三种模式都值得参考 |
| Docker 资源控制 | RD-Agent、AutoResearchClaw | RD-Agent 的 Docker SDK 参数更成熟，AutoResearchClaw 的网络策略更贴近科研安全 |
| 网络策略 | AutoResearchClaw | 支持 `none/setup_only/pip_only/full` 思路 |
| 远端执行 | AutoResearchClaw、deer-flow | SSH remote 和 remote provisioner 都有参考价值 |
| 持久会话 | SWE-agent、deer-flow、AI-Researcher | SWE-agent 的 session 最适合代码调试，AI-Researcher 的 TCP shell 安全较弱 |
| 工具调用审计 | deer-flow、AutoResearchClaw | bash 审计和 AST validator 可组合 |
| 实验指标采集 | AutoResearchClaw | `SandboxResult.metrics` 和 partial result 思路最接近科研需求 |
| 缓存复现 | RD-Agent | cache hash、日志、环境配置较成熟 |
| 轻量 Python 执行 | DeepResearch/Tongyi | SandboxFusion 模式适合短代码工具 |
| 科研流程 | AI-Scientist v1/v2 | idea、实验、绘图、报告流程可参考，但沙箱弱 |

## 关键设计取舍

### 1. LocalSandbox 只能用于开发

deer-flow 的 LocalSandbox 有路径映射和只读挂载，但项目自己也明确它不是安全边界。AI-Scientist 的本地 subprocess、RD-Agent LocalEnv、Auto-Deep-Research LocalEnv 都类似。

我们的系统中，LocalSandbox 应仅用于：

- 开发调试；
- 可信脚本；
- 只读分析；
- 单机演示。

Agent 生成代码的正式执行应默认走 Docker/Kubernetes。

### 2. Docker 不是自动安全

AI-Researcher 和 Auto-Deep-Research 都用了 Docker，但仍然存在 root、裸端口、无限制 shell、无资源限制、无网络策略等问题。Docker 只有在正确配置时才是沙箱。

推荐 Docker 最小策略：

```bash
docker run --rm \
  --user nonroot \
  --network none \
  --memory 16g \
  --cpus 8 \
  --pids-limit 512 \
  --shm-size 2g \
  -v workspace:/workspace:rw \
  -v data:/data:ro \
  -w /workspace \
  image \
  python run.py
```

### 3. 实验运行应区别于交互式命令

SWE-agent/deer-flow 的 bash 工具适合交互式调试，但正式实验应走 `run_project` 或 batch job：

- 明确入口；
- 明确 timeout；
- 明确环境变量；
- 明确输出目录；
- 明确 metrics；
- 明确 artifact；
- 执行后清理。

AutoResearchClaw 的 `run_project` 是较好的参考。

### 4. 工业工程需要特殊资源治理

通用科研 Agent 沙箱通常只关注 Python/Docker/GPU。工业工程还需要：

- 求解器许可证；
- 许可证服务器访问；
- 多线程求解控制；
- 大规模数据只读挂载；
- 解文件和日志保留；
- 可行解中断保存；
- 商业软件镜像；
- 任务排队和资源计费。

这些不是任何一个仓库完全覆盖的，需要在我们的系统里补齐。

## 推荐组合架构

最推荐的组合是：

```text
deer-flow/SWE-agent 风格控制面
  + AutoResearchClaw 风格 run_project / network policy / metrics
  + RD-Agent 风格 Docker SDK / GPU / CPU / memory / cache
  + DeepResearch 风格轻量 Python interpreter
  + 工业工程专用 solver/data/license/artifact 策略
```

对应模块可设计为：

- `SandboxProvider`：负责 acquire/release/destroy；
- `DockerExperimentSandbox`：正式实验执行；
- `InteractiveSessionSandbox`：Agent 调试和代码修改；
- `PythonInterpreterSandbox`：短代码计算；
- `RemoteBatchSandbox`：大规模远端求解；
- `ArtifactCollector`：采集 `metrics.json`、日志、解文件、图表；
- `PolicyEngine`：网络、文件、命令、资源和许可证策略；
- `AuditLogger`：记录 Agent 工具调用和沙箱事件。

## 对每个仓库的最终建议

### AutoResearchClaw

作为主参考。重点吸收：

- `SandboxProtocol`；
- Docker/network policy；
- entry point 校验；
- `SandboxResult.metrics`；
- Agentic Docker。

### RD-Agent

作为执行器参考。重点吸收：

- Docker SDK；
- resource config；
- GPU device request；
- cache；
- log streaming；
- cleanup。

### deer-flow

作为控制面参考。重点吸收：

- `SandboxProvider`；
- middleware；
- Local/Aio/Remote backend；
- warm pool；
- bash audit。

### SWE-agent

作为交互式编码 Agent 参考。重点吸收：

- deployment runtime；
- persistent bash session；
- repo reset；
- tool install/reset；
- timeout 熔断。

### DeepResearch/Tongyi

作为轻量 Python 工具参考。重点吸收：

- external sandbox endpoint；
- narrow Python interface；
- short timeout。

### AI-Scientist v1/v2

作为科研循环参考，不作为沙箱参考。重点吸收：

- idea -> experiment -> plot -> report；
- error feedback；
- v2 的子进程解释器和中断逻辑。

### AI-Researcher / Auto-Deep-Research

作为持久容器和流式输出参考。不要直接复用其安全模型。重点吸收：

- JSON streaming；
- persistent workspace；
- task-level container naming；
- port/workspace allocation。

## 最终落地优先级

建议按三步实现：

1. 先做 Docker 实验沙箱：
   - 每个任务独立 workspace；
   - AutoResearchClaw 风格 `run_project`；
   - RD-Agent 风格资源限制；
   - 默认 `setup_only` 或 `none` 网络；
   - 统一 metrics/artifacts。

2. 再做 Agent 会话沙箱：
   - deer-flow/SWE-agent 风格 provider 和 session；
   - bash/read/write/grep 工具；
   - 命令审计；
   - 连续 timeout 熔断。

3. 最后做生产调度：
   - Kubernetes/Slurm/SSH provider；
   - warm pool；
   - 商业求解器许可证；
   - 数据访问审计；
   - 镜像签名和缓存复现。

一句话总结：AutoResearchClaw 给出最接近科研沙箱的整体形状，RD-Agent 给出可靠执行器，deer-flow 和 SWE-agent 给出 Agent 生命周期与工具接口，DeepResearch 给出轻量解释器模式；我们的工业工程科研机器应把这些优点组合起来，而不是直接照搬某一个仓库。
